- A sovereign mission operating system

- A hub of plugins
- A pretty interface with no consequence
- A stack zoo (many stacks with no spine)

- A product that rewards fragmentation

| Category | Rule |

## BENCHMARK MASTER LAW

All stacks are benchmarked by the same method. This law governs what each stack absorbs and what each stack rejects.

### Absorb
- The best solved value from external pioneers in this domain
- The strongest working structural patterns
- The deepest user-pain insights already validated in the market
- The inevitable future direction of this domain

### Reject
- Fragmentation (multiple tools doing one job)
- Tool hopping (forcing operators to switch context)
- SaaS clutter (features that exist to justify pricing, not to serve operators)
- Wrapper logic (AI-flavored shells over existing products)
- Decorative AI (AI that appears but does not produce consequence)
- Dead surfaces (anything that cannot be clicked, navigated, or functionally triggered)
- Infrastructure overexposure (surfaces that make operators manage infrastructure instead of executing missions)
- Anything anti-Ruberra (anything that moves toward the CANNOT list)

### Per-Stack Benchmark Encoding
Each stack carries its own benchmark record encoding:
- What this stack replaces (the external tools, patterns, or costs it eliminates)
- What this stack learns from (the strongest existing work in this domain)
- What this stack rejects (anti-patterns specific to this domain)
- What pain this stack resolves (the operator-felt problem this stack ends)

The benchmark encoding for each stack is maintained in `src/app/dna/stack-registry.ts`.

## ANTI-DRIFT LAW

Drift is the slow movement of Ruberra toward what it must never become. Drift is often invisible — it accumulates in small decisions, borrowed patterns, and features that seem reasonable in isolation.

**Drift signals to watch:**
- A new surface is added that has no consequence
- A feature is added because "every product has this"
- UI is made to look like an external product (Notion, Linear, Vercel, etc.)
- A model or tool dependency is added without questioning whether Ruberra should own this
- Pioneer output is merged without a cascade law check
- The word "dashboard" is used to describe any Ruberra surface
- A feature is described in terms of what it looks like rather than what consequence it produces

**Anti-drift enforcement:**
- Every pioneer must run the identity filter before proposing any surface
- Every merge must pass the cascade law before landing on main
- The Antigravity ledger must record every deviation from this law
- Claude conducts order and kills drift before it accumulates
