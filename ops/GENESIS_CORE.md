# RUBERRA GENESIS CORE

## Purpose

Ruberra Genesis Core is not a second product, not a second repo, and not a second UI.
It is the single operating doctrine for building Ruberra without drift.

Ruberra Product = the living MVP.
Genesis Core = the governing method used to conceive, command, remember, and build the product.

## Primary Law

No meta-system may exist unless it produces direct product progress.
No doctrine may survive if it does not sharpen truth, command, creation, build, or memory.

## The Five Functional Centers

### 1. Truth
What Ruberra is.
What Ruberra is not.
What has already been decided.
What cannot drift.

### 2. Command
The active wave.
The next move.
The currently required role.
The acceptance condition.
The blocker map.

### 3. Creation
Where ideas become architectures.
Where architectures become blocks.
Where blocks become executable work.

### 4. Build
The live repository.
The branches.
The files touched.
The implementation blocks.
The merge path.

### 5. Memory
The retained history.
The resume anchors.
The canon promotions.
The prior failures.
The continuity of work.

## What Genesis Core Is Not

- not a site
- not a dashboard
- not a notes app
- not a second MVP
- not an orchestration fantasy disconnected from shipping

## What It Does

Genesis Core governs how Ruberra advances:
- one repo
- one product
- one canon
- one active wave at a time
- one next move at a time
- no decorative complexity

## Role Mapping

### Ivan
Sovereign command. Final truth compression. Final direction.

### Architect
Protects topology, chamber placement, state, memory, and anti-fragmentation.

### Visual
Protects visual truth, hierarchy, chamber gravity, calm density, and anti-generic integrity.

### Builder
Turns direction into branches, files, implementation blocks, and mergeable work.

### Audit
Protects canon, detects regressions, blocks false completion, and enforces acceptance.

## Repository Doctrine

The repository must tell one truth only:
- `src/ruberra/` is the living organism
- `src/app/` is legacy quarantine until removed or selectively harvested

Any document, branch, or work item that contradicts this is drift.

## Product Doctrine

The product is Ruberra.
The product is not the doctrine.
The doctrine exists only to make the product stronger, truer, and more buildable.

## Execution Discipline

Every wave must produce both:
1. product advancement
2. operating clarity

If a wave produces only doctrine, it is incomplete.
If a wave produces only code but weakens canon, it is invalid.

## The Test

Genesis Core is functioning only when Ruberra becomes:
- clearer
- more sovereign
- less generic
- more buildable
- more memory-bearing
- more consequence-driven
- easier to continue tomorrow without re-framing from zero
