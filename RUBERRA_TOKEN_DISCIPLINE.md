# RUBERRA — TOKEN DISCIPLINE
MANDATORY PRE-TASK READ · SOVEREIGN COMPRESSION LAW · NO WASTE · NO DRIFT


This file governs all prompt input and agent output inside the Ruberra workspace.
It is not a suggestion. It is operational law.

Read it before opening any task.

━━━━━━━━━━━━━━━━━━
1. CORE TOKEN LAW
━━━━━━━━━━━━━━━━━━

Every token must earn its place.
If it does not carry consequence, remove it.

**Banned token patterns:**
- Preamble ("Sure, here is…", "Of course, I'll…", "Great question…")
- Re-explaining what was just stated
- Option menus ("Here are some possibilities:")
- Hedging without evidence ("might", "could potentially", "in some cases")
- Decorative padding that fills space without advancing state
- Restating the question before answering
- Closing affirmations ("Let me know if you need anything else")

**Required token patterns:**
- Podium answer first — the strongest, most direct answer leads
- Evidence follows the answer — never precedes it
- One direction — no branching into lateral variants unless unavoidable
- Compress settled truths — reference by name, do not re-derive
- Stop when done — no trailing summaries

━━━━━━━━━━━━━━━━━━
2. PROMPT COMPRESSION LAW
━━━━━━━━━━━━━━━━━━

All prompts submitted to agents must be compressed before dispatch.

**Compression rules:**
- Strip filler words and pleasantries
- State the target state, not the process to get there
- Name the affected file, component, or system directly
- One clear output requirement per prompt block
- If context is already in a sovereign file (constitution, repo self-knowledge, stack tracker), reference it — do not repeat it inline

**Prompt structure:**
```
[TARGET] — what must be true after this task
[SCOPE] — file / component / stack / system
[CONSTRAINT] — what must not change
[OUTPUT] — exact deliverable
```

━━━━━━━━━━━━━━━━━━
3. OUTPUT COMPRESSION LAW
━━━━━━━━━━━━━━━━━━

All agent outputs must follow this format:

**For code tasks:**
- Changed files only — no unchanged context blocks
- No explanatory prose before the code unless the change is non-obvious
- Inline comment only if the logic would be opaque without it
- After code: one-line consequence statement

**For analysis tasks:**
- Finding first — not background
- Evidence inline — not in a separate section
- Recommendation last — one sentence, one direction

**For documentation tasks:**
- Title → Law → Examples (if needed)
- No introductory paragraph
- No "as you can see" or "it is important to note"

**For status tasks:**
- Three fields maximum: FILE · COMMIT STATUS · BLOCKER
- Each field: one line
- No narrative

━━━━━━━━━━━━━━━━━━
4. FORBIDDEN OUTPUT STRUCTURES
━━━━━━━━━━━━━━━━━━

Never produce:
- Numbered lists of equal alternatives with no declared winner
- Bullet trees deeper than two levels without structural necessity
- Section headers for sections containing only one sentence
- "Summary" sections that repeat what was just said
- "Next steps" that list things not committed to
- Any response that treats uncertainty as a reason to expand rather than compress

━━━━━━━━━━━━━━━━━━
5. REFERENCE DISCIPLINE
━━━━━━━━━━━━━━━━━━

Sovereign files are canonical. Reference them; do not reproduce them.

| File | What it owns |
|------|-------------|
| `RUBERRA_WORKSPACE_CONSTITUTION.md` | Identity, organs, stack order, visual law, anti-drift |
| `RUBERRA_REPO_SELF_KNOWLEDGE.md` | Current state of each stack — closed / partial / open |
| `RUBERRA_STACK_CLOSURE_TRACKER.md` | Stack closure audit log |
| `CLAUDE.md` | Project memory, mandatory pre-task reads |
| `RUBERRA_TOKEN_DISCIPLINE.md` | This file — token and output compression law |

When a fact is already in a sovereign file, write:  
`→ see [FILENAME], [Section]`  
Never re-derive it inline.

━━━━━━━━━━━━━━━━━━
6. ENFORCEMENT
━━━━━━━━━━━━━━━━━━

Any output that violates this law is non-compliant.
Non-compliant outputs must be regenerated before they are acted upon.

Violations by class:
- **Class 1 (Waste):** Preamble, padding, re-explanation, trailing affirmations. Regenerate immediately.
- **Class 2 (Drift):** Option menus, lateral branching, hedging without evidence. Strip and redirect.
- **Class 3 (Noise):** Decorative structure, empty sections, summary repetition. Remove before commit.

━━━━━━━━━━━━━━━━━━
7. STATUS OUTPUT FORMAT
━━━━━━━━━━━━━━━━━━

All status responses (file ops, commits, deployments) return exactly:

```
FILE CREATED    — [filename] at [path]
COMMIT STATUS   — [committed / pending / failed] · [commit hash if committed]
BLOCKER         — [none / description of blocker]
```

No additional lines.

━━━━━━━━━━━━━━━━━━

*One token. One consequence. No waste.*
